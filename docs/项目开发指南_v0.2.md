# 项目开发指南 v0.4

本指南用于指导 `medical-ai-knowledge-platform` 的后续开发。项目目标不是简单上传作者的个人知识库，也不是提供一个空的 Obsidian 模板，而是开源一套可复刻的“知识库生成与执行系统”。

用户从 GitHub 下载后，应能逐步搭建出与作者本地原型类似的系统：有 raw 原始资料层、有 wiki 结构化知识层、有 Obsidian/Claudian/LLM 整理方法、有 Codex/Claude Code 可调用的执行平台，并能通过示例学会继续扩展自己的知识库。

## 1. 项目目的

本项目要开源的是一套方法和框架：

```text
原始资料 raw
→ LLM/Claudian 整理
→ 结构化 wiki
→ Codex / Claude Code 检索调用
→ 执行平台创建任务
→ R/Python 运行
→ 结果归档
→ 候选知识沉淀回 wiki 或 skill
```

它不是：

- 不是作者个人完整知识库的公开备份。
- 不是只包含 README 和空目录的模板。
- 不是只给人看的静态 Markdown 文档库。
- 不是只跑脚本的执行器。

它应该是：

- 一个可初始化的 Obsidian 知识库框架。
- 一套 raw 资料整理成 wiki 的 LLM 工作流。
- 一组可公开的 raw/wiki 示例。
- 一个能让 Codex/Claude Code 检索并调用知识库的执行平台。
- 一个可持续演化的科研 AI 操作系统雏形。

## 2. 用户下载后的理想体验

用户从 GitHub 下载项目后，应该可以这样使用：

1. 运行 `bootstrap.ps1`。
2. 选择本地安装目录。
3. 自动生成本地平台：

```text
<YOUR_INSTALL_DIR>/
├── my-vault/
├── execution-platform/
├── my-skills/
└── workspace.code-workspace
```

4. 用 VS Code 打开 `workspace.code-workspace`。
5. 用 Obsidian 打开 `my-vault/`。
6. 在 `my-vault/raw/` 中看到示例原始项目。
7. 在 `my-vault/wiki/` 中看到整理后的知识单元。
8. 阅读 `CLAUDE.md` / `AGENTS.md`，理解 AI 应如何使用这个知识库。
9. 通过 Codex 或 Claude Code 调用 `execution-platform/platform-core/scripts/search_kb.py` 检索知识模板。
10. 创建第一个分析任务，运行 demo，生成输出与归档报告。
11. 把自己的论文、数据、代码放入 `my-vault/raw/`。
12. 按指南用 LLM/Claudian 把 raw 整理成新的 wiki 知识单元。
13. 后续把高复用流程沉淀成自己的 `my-skills/`。

目标是让用户不是“拿到一个成品”，而是“学会搭建并演化自己的同类系统”。

## 3. 系统分层

GitHub 版应包含四层能力。

### 3.1 知识库层

对应作者本地知识库原型：

```text
本地医学科研知识库原型/
```

GitHub 版对应：

```text
starter-vault/
```

职责：

- 管理 raw 原始资料。
- 管理 wiki 结构化知识。
- 提供 Obsidian 可读的知识库结构。
- 提供 `CLAUDE.md` / `AGENTS.md`，告诉 AI 如何读、整理、使用知识。

### 3.2 知识整理层

对应作者通过 Obsidian + Claudian + Claude 形成的整理方法。

职责：

- 规定 raw 放什么。
- 规定 wiki 怎么生成。
- 规定 README、manifest、code、data、output 如何组织。
- 规定如何更新 `wiki/index.md` 与 `wiki/log.md`。
- 规定如何补全 concepts，避免空概念节点。

这一层必须文档化，不能只存在于作者习惯里。

### 3.3 执行平台层

对应作者本地执行平台原型：

```text
医学科研AI执行平台/
```

GitHub 版对应：

```text
platform-core/
```

职责：

- 检索知识库。
- 校验知识库。
- 创建分析任务目录。
- 复用 wiki 中的模板代码。
- 运行 R/Python。
- 记录 `run_manifest.yaml`。
- 生成结果摘要和知识沉淀候选。

### 3.4 AI 调用层

支持 Codex、Claude Code 或其他 MCP 客户端调用知识库与执行平台。

调用方式至少包括：

- 文件规则：读取 `CLAUDE.md` / `AGENTS.md`。
- 终端脚本：运行 `search_kb.py`、`create_analysis_project.py` 等。
- MCP 工具：调用 `knowledge.search`、`analysis.create_task` 等。

## 5. starter-vault 设计

`starter-vault` 应尽量复刻作者本地知识库结构，但只放可公开内容。

```text
starter-vault/
├── CLAUDE.md
├── AGENTS.md
├── 首页.md
├── raw/
│   ├── README.md
│   ├── papers/
│   │   └── README.md
│   └── analyses/
│       ├── README.md
│       ├── 连续数据的Meta分析/
│       ├── 网状Meta分析/
│       ├── 机器学习及R应用/
│       └── ...
├── wiki/
│   ├── index.md
│   ├── log.md
│   ├── analyses/
│   ├── concepts/
│   ├── sources/
│   └── synthesis/
└── _templates/
```

### 5.1 raw 的作用

`raw/` 是原始资料层。

它让用户看到真实资料进入知识库前是什么样：

- 原始 R/Python 脚本。
- 原始示例数据。
- 原始输出图或输出说明。
- 学习笔记。
- 来源说明。

但它不是随意上传个人资料的地方。GitHub 版 raw 只能放可公开内容。

### 5.2 wiki 的作用

`wiki/` 是 LLM 可查阅、可复用的结构化知识层。

每个 `wiki/analyses/<项目>/` 应包含：

```text
README.md
manifest.yaml
code.R 或 code.py
data/
output/
```

`README.md` 给人和 LLM 深度理解，`manifest.yaml` 给脚本和 MCP 检索。

## 6. raw -> wiki 整理工作流

这是项目最重要的工作流之一。

用户添加新分析项目时：

```text
把原始材料放入 raw/analyses/<项目名>/
```

LLM/Claudian/Codex/Claude Code 应执行：

1. 读取 raw 项目的 README、SOURCE、代码、数据、输出和 notes。
2. 判断分析类型、方法、输入格式、输出类型和适用场景。
3. 在 `wiki/analyses/<项目名>/` 下创建结构化知识单元。
4. 编写 `README.md`，至少包含：
   - 问题背景。
   - 适用场景。
   - 数据格式要求。
   - 代码逻辑。
   - 输出解释。
   - 常见错误。
   - 新数据适配指南。
   - 对应 raw 项目位置。
5. 编写 `manifest.yaml`，用于程序检索。
6. 清理 `code.R` 或 `code.py`，保证相对路径可运行。
7. 放置可公开示例数据或数据说明。
8. 放置输出说明，必要时保留少量可公开示例输出。
9. 更新 `wiki/index.md`。
10. 更新 `wiki/log.md`。
11. 如出现新概念，补充 `wiki/concepts/`，避免空白概念节点。

这套流程应写入：

- `starter-vault/CLAUDE.md`
- `starter-vault/AGENTS.md`
- `docs/Claudian整理工作流.md`
- `docs/wiki知识单元创建规范.md`

## 7. 可公开示例设计

GitHub 版不能只给空目录。它应包含一批可公开的“原始项目 + wiki 知识单元”示例。

优先迁移：

| raw 原始项目 | wiki 知识单元 | 优先级 |
|---|---|---:|
| `raw/analyses/连续数据的Meta分析/` | `wiki/analyses/连续数据的Meta分析/` | P0 |
| `raw/analyses/网状Meta分析/` | `wiki/analyses/网状Meta分析/` | P1 |
| `raw/analyses/机器学习及R应用/` | `wiki/analyses/机器学习-线性模型/` 等 | P1 |
| `raw/analyses/二分类数据的Meta分析/` | `wiki/analyses/二分类数据的Meta分析/` | P2 |
| `raw/analyses/单个率的Meta分析/` | `wiki/analyses/单个率的Meta分析/` | P2 |
| `raw/analyses/效应值的Meta分析/` | `wiki/analyses/效应值的Meta分析/` | P2 |
| `raw/analyses/相关系数的Meta分析/` | `wiki/analyses/相关系数的Meta分析/` | P3 |
| `raw/analyses/剂量反应关系Meta分析/` | `wiki/analyses/剂量反应关系Meta分析/` | P3 |

迁移原则：

- 可公开代码可以保留。
- 可公开数据可以保留。
- 不确定授权的数据应替换成合成数据。
- 未授权 PDF/PPT 不上传。
- 大量输出图不上传，优先用 `output/README.md` 说明。
- 每个 raw 项目必须有 `SOURCE.md`。

## 8. 执行平台设计

执行平台必须能让 Codex/Claude Code 调用知识库。

核心脚本：

```text
platform-core/scripts/
├── search_kb.py
├── validate_kb.py
├── create_analysis_project.py
├── run_r_task.py
└── archive_result.py
```

标准执行流程：

```text
用户提出分析任务
→ AI 读取 my-vault/CLAUDE.md 或 AGENTS.md
→ AI 检索 wiki/index.md 和 manifest.yaml
→ search_kb.py 返回候选模板
→ create_analysis_project.py 创建 projects/<task>
→ AI 复制/改写模板代码
→ run_r_task.py 或 runner 执行
→ archive_result.py 归档
→ 生成 knowledge_candidate.md
→ 用户确认后再沉淀回 wiki 或 skill
```

标准任务目录：

```text
execution-platform/projects/<task>/
├── task.md
├── run_manifest.yaml
├── data/
├── src/
├── output/
├── report/
└── logs/
```

执行平台的边界：

- 可以写 `projects/`、`outputs/`、`logs/`。
- 不直接覆盖 `my-vault/wiki/` 中的标准模板。
- 只有用户确认后，才把候选知识沉淀回 `wiki/` 或 `my-skills/`。

## 9. Codex / Claude Code / MCP 接入

GitHub 版必须让 AI 能调用知识库，而不仅是人手动读文件。

### 9.1 规则文件

`starter-vault/CLAUDE.md` 和 `starter-vault/AGENTS.md` 应说明：

- 先读 `wiki/index.md`。
- 再检索 `wiki/analyses/*/manifest.yaml`。
- 能复用模板时不从零生成。
- 任务写入 `execution-platform/projects/`。
- 输出、日志、报告写入任务目录。
- 候选知识经用户确认后再沉淀。

### 9.2 脚本调用

AI 可以调用：

```powershell
python .\execution-platform\platform-core\scripts\search_kb.py "连续变量 Meta 分析" --json
python .\execution-platform\platform-core\scripts\validate_kb.py --json
python .\execution-platform\platform-core\scripts\create_analysis_project.py "demo task"
```

### 9.3 MCP 调用

保留 MCP 原型：

```text
platform-core/mcp/medical_research_platform/
```

工具：

- `knowledge.search`
- `knowledge.validate`
- `analysis.create_task`
- `analysis.run_r`
- `archive.result`

MCP 配置必须使用用户本地安装目录和 `MAKP_PATHS_CONFIG`，不能写死作者本地路径。

## 10. bootstrap 设计

`bootstrap.ps1` 要完成的不只是复制文件。

它应完成：

1. 询问安装目录。
2. 创建本地根目录。
3. 复制 `starter-vault` 到 `my-vault`。
4. 复制 `platform-core` 到 `execution-platform/platform-core`。
5. 创建 `projects/`、`outputs/`、`logs/`。
6. 创建 `my-skills/`。
7. 生成 `execution-platform/config/paths.yaml`。
8. 生成 `workspace.code-workspace`，同时挂载 `my-vault` 和 `execution-platform`。
9. 提示用户用 Obsidian 打开 `my-vault`。
10. 提示用户用 Codex/Claude Code 阅读 `CLAUDE.md` / `AGENTS.md`。
11. 提示用户运行 demo 检索和 demo 任务。

bootstrap 结束提示应清楚说明：

```text
1. Open workspace.code-workspace
2. Open my-vault in Obsidian
3. Read my-vault/首页.md
4. Ask Codex/Claude Code to read CLAUDE.md or AGENTS.md
5. Run knowledge search demo
6. Create and run first analysis task
```

## 11. 开发路线

### v0.1 双核心骨架

目标：GitHub 版同时具备知识库和执行平台。

任务：

- 完成 `starter-vault` 基础结构。
- 完成 `platform-core` 基础脚本。
- bootstrap 生成 `my-vault` 和 `execution-platform`。
- workspace 同时挂载两者。
- README 说明双核心设计。

验收：

- 新用户可以初始化本地平台。
- Codex/Claude Code 可以看到知识库和执行平台。
- `search_kb.py` 可以检索 starter wiki。

### v0.2 raw/wiki 同构样例

目标：让 GitHub 版能展示作者知识库的成长方式。

任务：

- 建立 `starter-vault/raw/analyses`。
- 迁移连续数据 Meta 的 raw 和 wiki。
- 迁移网状 Meta 的 raw 和 wiki。
- 迁移机器学习 raw 总目录和至少一个 wiki 子模板。
- 更新 `wiki/index.md`。

验收：

- 用户能看到 raw 原始项目。
- 用户能看到整理后的 wiki 知识单元。
- raw 和 wiki 之间有明确对应关系。

### v0.3 执行闭环

目标：至少一个模板能端到端运行。

任务：

- 连续数据 Meta 可以被 `search_kb.py` 检索。
- 能创建任务目录。
- 能运行 R 脚本。
- 能归档结果。
- 能生成 `knowledge_candidate.md`。

验收：

- 用户能通过 Codex/Claude Code 完成一次 demo 分析任务。

### v0.4 Claudian/LLM 整理指南

目标：用户能学会把自己的 raw 整理成 wiki。

任务：

- 写 `docs/Claudian整理工作流.md`。
- 写 `docs/raw资料整理规范.md`。
- 写 `docs/wiki知识单元创建规范.md`。
- 更新 `starter-vault/CLAUDE.md`。
- 更新 `starter-vault/AGENTS.md`。

验收：

- 用户知道如何导入新 raw 项目。
- 用户知道如何让 LLM 编译成 wiki。
- 用户知道如何更新 index 和 log。

### v0.5 MCP/Codex/Claude Code 工具化

目标：AI 可以通过工具接口稳定调用平台能力。

任务：

- 完善 MCP server。
- 完善 `mcp_config.example.json`。
- 写 `docs/Codex使用说明.md`。
- 写 `docs/ClaudeCode使用说明.md`。
- 加入 MCP smoke test。

验收：

- `knowledge.search` 可用。
- `knowledge.validate` 可用。
- `analysis.create_task` 可用。
- `analysis.run_r` 可用。
- `archive.result` 可用。

## 12. GitHub 开源边界

默认不上传：

- 真实患者数据。
- 真实课题私有数据。
- 未授权论文全文。
- 未授权书籍 PDF/PPT。
- 本地运行 projects。
- logs、outputs。
- `paths.yaml`。
- `.Rproj.user/`。
- `.Rhistory`。
- 个人账户路径和缓存。

可以上传：

- 框架代码。
- AI 规则文件。
- 作者原创文档。
- 可公开 raw 示例。
- 可公开 wiki 模板。
- 合成数据。
- 明确许可的数据。
- 可运行 demo。

每个来自外部来源的 raw 示例都应有：

```text
SOURCE.md
```

说明来源、许可、是否允许再分发、是否做过脱敏或合成替换。

